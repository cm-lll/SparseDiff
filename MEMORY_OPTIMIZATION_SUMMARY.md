# 内存优化总结

## 发现的问题

### 1. 损失计算中的内存占用 ⚠️

在 `compute_Lt` 中创建了大量中间张量：
- `prob_true_E = torch.zeros_like(E)` - (bs, n, n, de) - **大张量**
- `prob_pred_E = torch.zeros_like(pred_probs_E)` - (bs, n, n, de) - **大张量**
- 每个关系族的 one-hot 编码
- 每个关系族的局部状态张量
- 映射后的全局状态张量

**对于大图（n很大），这些张量会占用大量内存。**

### 2. 验证步骤中的密集化 ⚠️

在 `validation_step` 中：
- `densify_noisy_data` 将稀疏数据转换为密集格式
- `E_t` 变成 `(bs, n, n, de)` 的密集张量
- 对于大图，这会占用大量内存

### 3. 中间张量未及时释放 ⚠️

创建了很多中间张量但没有及时释放。

## 已应用的优化

### ✅ 1. 减少中间张量创建

- 直接填充到全局概率矩阵，避免创建 `prob_true_E_b_fam_global` 和 `prob_pred_E_b_fam_global`
- 使用 `del` 释放不再需要的中间张量

### ✅ 2. 内存管理

- 在循环中及时释放中间张量
- 减少不必要的张量复制

## 建议的进一步优化

### 1. 减小 batch_size（最简单有效）

```bash
train.batch_size=4  # 或更小
```

### 2. 使用梯度检查点（如果支持）

可以减少前向传播的内存占用。

### 3. 优化验证步骤

- 可以考虑在验证时使用更小的图
- 或者禁用某些特征计算

## 当前状态

- ✅ 已优化损失计算中的内存使用
- ⚠️ 验证步骤中的密集化是必要的（用于计算损失）
- ⚠️ 建议减小 batch_size 以避免 OOM
